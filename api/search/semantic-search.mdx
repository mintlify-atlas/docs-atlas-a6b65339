## Overview

Semantic search uses LanceDB vector embeddings powered by Google Gemini (`gemini-embedding-001`, 3072 dimensions) to find conceptually relevant content, not just keyword matches.

<Note>
  **Hybrid Search Architecture**: Internally, `semantic_search` uses a hybrid approach combining vector similarity with keyword matching (BM25) and Reciprocal Rank Fusion (RRF). This provides superior results over pure vector or keyword search alone. The underlying implementation is in `hive_commons/src/hive_commons/vector_store.py`.
</Note>

**When to use**: Natural language queries, conceptual searches, when exact keywords are unknown, or when you need synonym-aware search.

## Function Signature

```python
async def semantic_search(query: str, top_k: int = 5, stack: str = "") -> str
```

## Parameters

<ParamField path="query" type="string" required>
  Natural language query describing what you're looking for. Automatically expanded for better matching.
  
  **Examples**: 
  - "how to implement RAG pipelines"
  - "best practices for React state management"
  - "authentication patterns in FastAPI"
</ParamField>

<ParamField path="top_k" type="integer" default="5">
  Number of results to return. Internally fetches `top_k * 2` when using stack filtering for better relevance.
  
  **Range**: 1-20 recommended (higher values increase API latency)
</ParamField>

<ParamField path="stack" type="string" default="">
  Comma-separated technology stack filter. Results mentioning these technologies are boosted in ranking.
  
  **Format**: `"python,fastapi"` or `"react,typescript,nextjs"`
  
  **Effect**: Retrieves 2x results, then re-ranks based on stack token matches
</ParamField>

## Return Format

Returns formatted markdown string with semantic matches:

```markdown
Found 5 semantic matches for 'event sourcing CQRS':

### 1. Score: 0.847
Source: knowledge/skills/RAG_SYSTEMS_2026_SOTA.md
```
CQRS pattern separates read and write models, enabling better scalability...
```

### 2. Score: 0.792
Source: knowledge/EUREKA/EUREKA_HYBRID_SEARCH.md
```
Event sourcing stores state changes as a sequence of events...
```
```

If no matches:
```
No semantic matches for: {query}
```

```python
# Expanded: "testing unit test integration test e2e playwright vitest"

```

**Supported expansions**: caching, testing, deployment, security, performance, migration, API, database, auth, Docker, React, TypeScript, Astro, FastAPI, MCP, RAG, chunking, monitoring.

## Stack Filtering Logic

### How It Works

1. Fetch `top_k * 2` initial results (e.g., 10 for `top_k=5`)
2. Score each result by stack token matches:
   - +1 point: Stack token appears in source path or text
   - Sort by boost score (descending)
3. Return top `top_k` boosted results

### Example

```python
result = await semantic_search(
    query="caching strategies",
    top_k=3,
    stack="python,fastapi"
)

# Fetches 6 results, re-ranks:
# 1. Score: 0.82, Source: .../fastapi_caching_patterns.md (boost: 2)
# 2. Score: 0.89, Source: .../redis_cache_basics.md (boost: 1) 
# 3. Score: 0.91, Source: .../cache_theory.md (boost: 0)
#
# Even though #3 has highest semantic score,
# stack boost prioritizes FastAPI-specific content
```

## Performance

### Latency

- **First call**: ~800-1200ms (Gemini API embedding)
- **Cached query**: less than 50ms (embedding cache hit)
- **Cache TTL**: 5 minutes
- **LanceDB vector search**: ~20-50ms

### Query Embedding Cache

- **LRU cache**: 100 most recent queries
- **Key**: Expanded query text
- **Savings**: 1-2 seconds per cache hit

### Rate Limits

- **Gemini API**: Subject to Google API quotas
- **Failure handling**: Automatic retry once (1s delay)
- **Fallback**: Returns empty results if embedding fails

## Architecture Details

### Embedding Model

- **Model**: `gemini-embedding-001`
- **Dimensions**: 3072
- **Provider**: Google Gemini API
- **Quality**: State-of-the-art (2024-2026)

### Vector Store

- **Engine**: LanceDB
- **Index**: Vector + FTS (hybrid-ready)
- **Size**: 670K+ vectors
- **Storage**: Disk-based (persistent)

### Caching Layers

1. **Query embedding cache**: LRU, 5min TTL, 100 entries
2. **Chunk embedding cache**: In-memory, session-scoped
3. **LanceDB internal cache**: OS page cache
