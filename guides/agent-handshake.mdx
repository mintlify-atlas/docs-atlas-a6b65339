The `agent_handshake` tool tailors MidOS to your specific environment. Call it first when connecting to get optimized tool recommendations, skill matches, and context-aware configurations.

## Why Personalization Matters

MidOS serves diverse agents with different capabilities:
- **Models** vary in context windows (8K–2M tokens), tool support, and speed
- **Clients** have different features (hooks, memory, background agents)
- **Stacks** determine which skills and knowledge are relevant

Without personalization:
- ❌ Generic results with low relevance
- ❌ Irrelevant tools and skills shown
- ❌ No context budget optimization
- ❌ Missing model-specific tips

With personalization:
- ✅ Stack-filtered skills ranked by relevance
- ✅ Tools sorted by your use case
- ✅ Context budget matched to your window
- ✅ Guardrails adapted to your client's capabilities

## Quick Start

### Basic Handshake

Minimal information for quick setup:

```python
agent_handshake(
    model="claude-opus-4-6",
    client="claude-code"
)
```

### Full Handshake

Maximum personalization:

```python
agent_handshake(
    model="claude-opus-4-6",
    context_window=200000,
    client="claude-code",
    languages="python,typescript",
    frameworks="fastapi,react,postgresql",
    platform="macos",
    project_goal="building a SaaS analytics dashboard"
)
```

## Parameters Explained

### model (string)

Your AI model identifier. Used to:
- Detect context window
- Identify capabilities (tools, vision, structured output)
- Load model-specific tips and guardrails

**Supported model formats:**

<CodeGroup>
```python Claude models
# Full ID
model="claude-opus-4-6"
model="claude-sonnet-4"
model="claude-3.5-sonnet"

# Short name (auto-resolved)
model="opus"
model="sonnet"
```

```python OpenAI models
model="gpt-4-turbo"
model="gpt-4o"
model="o1"
model="o3-mini"
```

```python Google models
model="gemini-2.5-pro"
model="gemini-2.5-flash"
model="gemini-1.5-pro"
```

```python Open source
model="llama-3.3-70b"
model="qwen-2.5-72b"
model="deepseek-v3"
```
</CodeGroup>

**What happens:**
1. MidOS looks up model in catalog (150+ models)
2. Resolves context window, max output, capabilities
3. Assigns tier (small/medium/large context)
4. Loads model-specific tips

**If model not found:**
- Defaults to 128K context window
- Assumes tool support enabled
- Uses generic tips

### context_window (integer)

Explicit context window in tokens. Optional if model is recognized.

```python
# Auto-detect from model
model="claude-opus-4-6"  # → 200,000 tokens

# Override auto-detection
model="custom-model",
context_window=50000

# Unknown model with explicit window
model="my-fine-tuned-llama",
context_window=32000
```

**Used for:**
- Determining payload tier (small/medium/large)
- Truncating handshake response
- Recommending chunk vs full skill retrieval

**Tiers:**
- **Small** (up to 32K): Compact responses, keyword search preferred
- **Medium** (32K to 128K): Standard responses, balanced search
- **Large** (over 128K): Full responses with previews, semantic search preferred

### platform (string)

Your operating system. Used for:
- Platform-specific guardrails (Windows PowerShell, macOS paths)
- File path conventions
- Installation instructions

```python
platform="macos"   # or "linux", "windows"
```

### project_goal (string)

What you're building. Used for:
- Finding relevant knowledge chunks
- Prioritizing skills
- Generating proactive suggestions

```python
project_goal="building a real-time collaborative code editor"
```

**Good goals:**
- "building a SaaS analytics dashboard with real-time updates"
- "implementing event sourcing for an e-commerce platform"
- "creating a CLI tool for database migrations"

**Avoid generic goals:**
- ❌ "learning React" (too vague)
- ❌ "testing" (no context)
- ❌ "API" (be specific)

```python

    model="claude-opus-4-6",
```

## Context Window Optimization

### How It Works

1. **Resolve window**: From model catalog or explicit parameter
2. **Determine tier**: small (up to 32K), medium (32K–128K), large (over 128K)
3. **Truncate response**: Based on tier
   - Small: 2 skills, 3 guardrails, 2 tips
   - Medium: 5 skills, 5 guardrails, 5 tips
   - Large: 15 skills, full guardrails, full tips
4. **Recommend tools**: Prefer keyword search for small context

## Stack-Aware Recommendations

### Skill Ranking

Skills are scored by:

1. **Model recommendations** (score: 10): Skills explicitly recommended for your model
2. **Compatibility match** (score: 3 per match): `compatibility.json` languages/frameworks match
3. **Name match** (score: 1 per match): Skill name contains stack keyword

```python
# Example scoring for: languages="python", frameworks="fastapi"
fastapi_patterns:         score = 6  (2 name matches: "fastapi" + "python")
django_v5:                score = 3  (compatibility: python)
react_19_comprehensive:   score = 0  (no match)
```

### Tool Filtering

CLI profiles restrict tools by capability:

```json cli_profile.json (excerpt)
{
  "cursor": {
    "tool_restrictions": {
      "mode": "allowlist",
      "denied": ["pool_signal", "pool_status"],
      "explanation": "Cursor doesn't support background agents"
    }
  }
}
```

Handshake won't recommend `pool_signal` if you're using Cursor.

## Guardrails Adaptation

Guardrails change based on:

### Universal (Always Applied)

- Never hardcode secrets — use `os.getenv()` + `.env`
- Check indices before creating new ones
- Windows: use `;` not `&&` for command chaining
- Polling: max 10s between checks
- No duplication: synthesize, never scatter

### Model-Specific

**Small context models:**
- Keep tool calls concise
- Prefer search_knowledge over semantic_search
- Request summaries instead of full documents

**Models without tool support:**
- Use structured prompts instead of tool calls
- Request information in batch

### Client-Specific

**No hooks (Cursor, Cline):**
- Implement guardrails in prompts
- Use CLAUDE.md or project rules

**No memory (Cline):**
- Store context in project files
- Use episodic_store for key findings

**No background agents (Cursor, Cline):**
- Run tasks sequentially
- Batch operations to reduce round-trips

## Troubleshooting

<AccordionGroup>
  <Accordion title="Handshake returns generic config">
    **Cause**: Model or client not recognized.
    
    **Fix**: Use canonical names.
    
    ```python
    # Wrong
    model="claude opus", client="cc"
    
    # Correct
    model="claude-opus-4-6", client="claude-code"
    ```
  </Accordion>
  
  <Accordion title="No skills recommended">
    **Cause**: Stack keywords not matching catalog.
    
    **Fix**: Use full framework names.
    
    ```python
    # Wrong
    frameworks="js,pg,k8s"
    
    # Correct
    frameworks="javascript,postgresql,kubernetes"
    ```
  </Accordion>
  
  <Accordion title="Wrong context tier applied">
    **Cause**: Model catalog outdated or unknown model.
    
    **Fix**: Explicitly set context_window.
    
    ```python
    model="my-model",
    context_window=100000  # Force medium tier
    ```
  </Accordion>
  
  <Accordion title="Irrelevant tools shown">
    **Cause**: project_goal too generic or missing.
    
    **Fix**: Provide specific project description.
    
    ```python
    # Wrong
    project_goal="API"
    
    # Correct
    project_goal="building a GraphQL API for real-time chat with WebSockets"
    ```
  </Accordion>
</AccordionGroup>
