MidOS offers four search modes, each optimized for different query types. Understanding when to use each mode will dramatically improve your results.

## Quick Reference

| Search Mode | Best For | Speed | Precision |
|-------------|----------|-------|----------|
| `search_knowledge` | Exact terms, file names, code identifiers | ⚡ Fastest | High for exact matches |
| `semantic_search` | Natural language, concepts, synonyms | 🐢 Slowest | High for concepts |
| `hybrid_search` | Balanced queries, technical + conceptual | ⚡⚡ Fast | Highest overall |
| `smart_search` | Unknown query type, auto-detection | ⚡⚡ Fast | Auto-optimized |

## search_knowledge (Keyword)

Keyword search scans file names and content for exact term matches. Fast and deterministic.

### When to Use

<CardGroup cols={2}>
  <Card title="Known Terms" icon="tag">
    Searching for specific function names, error codes, or technical terms.
  </Card>
  <Card title="File Names" icon="file">
    Finding documents by title or path.
  </Card>
  <Card title="Code Patterns" icon="code">
    Looking for exact syntax or API calls.
  </Card>
  <Card title="Fast Lookups" icon="bolt">
    When speed matters more than semantic understanding.
  </Card>
</CardGroup>

### How It Works

1. Tokenizes your query into words
2. Matches against file names and content
3. Scores by word frequency and file name bonuses
4. Returns top N results sorted by score

### Example Queries

<CodeGroup>
```bash Good keyword queries
# Specific function name
search_knowledge("useOptimistic")

# Error code
search_knowledge("ECONNREFUSED")

# Framework API
search_knowledge("FastAPI dependency injection")

# File path fragment
search_knowledge("react/hooks")
```

```bash Poor keyword queries
# Too vague (use semantic instead)
search_knowledge("how to cache data")

# Natural language (use semantic instead)
search_knowledge("what is the best way to handle errors")

# Synonyms won't match
search_knowledge("speed up queries")  # Won't find "optimize performance"
```
</CodeGroup>

### Real Example

```python
# Query: Find FastAPI dependency patterns
results = search_knowledge("FastAPI dependency injection")
```

**Results:**
```json
[
  {
    "path": "knowledge/chunks/fastapi_dependencies.md",
    "score": 8,
    "preview": "FastAPI dependency injection uses Depends() to inject...",
    "size": 2456
  },
  {
    "path": "skills/fastapi_patterns/SKILL.md",
    "score": 6,
    "preview": "## Dependency Injection Patterns\n\nFastAPI's DI system..."
  }
]
```

### Stack Filtering

Keyword search doesn't have native stack filtering, but you can prefix queries:

```bash
# Implicit stack filtering via query terms
search_knowledge("React useEffect cleanup")
search_knowledge("Django ORM select_related")
search_knowledge("Rust ownership borrowing")
```

## semantic_search (Vector)

Semantic search uses Gemini embeddings (3072-d vectors) to find conceptually similar content, even if exact words don't match.

### When to Use

<CardGroup cols={2}>
  <Card title="Concepts" icon="brain">
    Searching by idea or pattern, not exact terms.
  </Card>
  <Card title="Natural Language" icon="message">
    Full sentence questions like "How do I...?"
  </Card>
  <Card title="Synonyms" icon="arrows-rotate">
    Query uses different words than the content (e.g., "optimize" vs "speed up").
  </Card>
  <Card title="Cross-Domain" icon="diagram-project">
    Finding patterns that apply across stacks (e.g., caching strategies).
  </Card>
</CardGroup>

### How It Works

1. Embeds your query using Gemini (`gemini-embedding-001`)
2. Searches 670K+ vectors in LanceDB
3. Returns top K by cosine similarity
4. Scores range 0.0–1.0 (higher = more similar)

### Example Queries

<CodeGroup>
```bash Good semantic queries
# Conceptual question
semantic_search("how to implement optimistic UI updates")

# Pattern-based
semantic_search("event sourcing patterns for microservices")

# Problem description
semantic_search("preventing race conditions in async code")

# Best practices
semantic_search("secure authentication with JWT tokens")
```

```bash Poor semantic queries
# Too broad (add context)
semantic_search("React")  # Too generic

# Exact code (use keyword instead)
semantic_search("import { useState } from 'react'")  # Keyword is better

# Very short (add more context)
semantic_search("caching")  # Say "Redis caching strategies" instead
```
</CodeGroup>

### Real Example

```python
# Query: How to optimize database queries (natural language)
results = semantic_search(
    "how to optimize slow database queries in production",
    top_k=5,
    stack="postgresql,django"
)
```

**Results:**
```json
[
  {
    "text": "PostgreSQL query optimization starts with EXPLAIN ANALYZE...",
    "source": "knowledge/chunks/postgresql_performance.md",
    "score": 0.847,
    "search_mode": "vector"
  },
  {
    "text": "Django ORM select_related and prefetch_related reduce N+1...",
    "source": "skills/django_v5/SKILL.md",
    "score": 0.792
  }
]
```

### Stack Filtering

Boost results for specific technologies:

```python
# Prioritize Python/FastAPI results
semantic_search(
    "implementing rate limiting",
    stack="python,fastapi"
)

# Frontend focus
semantic_search(
    "managing global state",
    stack="react,typescript"
)

# Multi-stack (comma-separated)
semantic_search(
    "authentication patterns",
    stack="nextjs,django,postgresql"
)
```

**How stack filtering works:**
1. Retrieves `top_k * 2` candidates
2. Scores each by stack token overlap in content
3. Re-ranks with stack boost applied
4. Returns top K results

## hybrid_search (Combined)

Hybrid search fuses keyword (BM25) and semantic (vector) results using Reciprocal Rank Fusion (RRF), then optionally reranks with a cross-encoder.

### When to Use

<CardGroup cols={2}>
  <Card title="Unknown Query Type" icon="question">
    Not sure if keyword or semantic is better.
  </Card>
  <Card title="Technical + Conceptual" icon="layer-group">
    Query has both exact terms and concepts (e.g., "FastAPI async patterns").
  </Card>
  <Card title="Best Overall Quality" icon="star">
    You want the highest precision regardless of speed.
  </Card>
  <Card title="Production Search" icon="globe">
    Default for user-facing search where quality matters.
  </Card>
</CardGroup>

### How It Works

<Steps>
  <Step title="Retrieve candidates">
    - Keyword search (BM25 FTS) returns top 30 results
    - Semantic search (vector) returns top 30 results
  </Step>

  <Step title="Fuse with RRF">
    Reciprocal Rank Fusion combines rankings:
    ```
    score(doc) = Σ [ 1 / (rank_in_list_i + 60) ]
    ```
    Documents appearing in both lists get higher scores.
  </Step>

  <Step title="Optional reranking">
    If `rerank=True`, applies cross-encoder model (`ms-marco-MiniLM-L-6-v2`) to re-score top candidates.
  </Step>

  <Step title="Return top K">
    Final results sorted by fused score.
  </Step>
</Steps>

### Example Queries

```python
# Default hybrid (no reranking)
hybrid_search("React 19 Server Components streaming")

# With reranking for maximum quality
hybrid_search(
    "PostgreSQL JSONB indexing performance",
    rerank=True
)

# Adjust vector/keyword balance with alpha
hybrid_search(
    "event sourcing CQRS patterns",
    alpha=0.7  # 70% vector, 30% keyword
)

# Stack-aware hybrid search
hybrid_search(
    "Redis caching strategies",
    stack="python,fastapi",
    rerank=True
)
```

### Alpha Parameter (Balance)

Control the vector/keyword balance:

- `alpha=0.0`: Pure keyword (BM25 only)
- `alpha=0.5`: Equal weight (default)
- `alpha=1.0`: Pure semantic (vector only)

```python
# Favor keyword matching (exact terms matter)
hybrid_search("useEffect cleanup", alpha=0.2)

# Favor semantic (conceptual match matters)
hybrid_search("best practices for API design", alpha=0.8)
```

### Reranking

Enable cross-encoder reranking for 5-10% precision gain:

```python
# Without reranking (faster)
results = hybrid_search("query", rerank=False)  # ~200ms

# With reranking (higher quality)
results = hybrid_search("query", rerank=True)   # ~400ms
```

**When to rerank:**
- User-facing search (quality over speed)
- Critical queries (architecture decisions)
- Final answer retrieval (not exploratory search)

**When to skip reranking:**
- Exploratory search (browsing)
- High-frequency automated queries
- Real-time autocomplete

## smart_search (Auto-Detect)

Smart search automatically selects the best mode based on query analysis. Currently defaults to `hybrid`, but future versions will include query classification.

### When to Use

<CardGroup cols={2}>
  <Card title="General Use" icon="wand-magic-sparkles">
    Let MidOS decide the best strategy.
  </Card>
  <Card title="Agent Tools" icon="robot">
    Default search for AI agents that generate varied queries.
  </Card>
  <Card title="User Search Bars" icon="magnifying-glass">
    End-user search where query type is unpredictable.
  </Card>
  <Card title="Prototyping" icon="flask">
    Testing queries without optimizing mode selection.
  </Card>
</CardGroup>

### How It Works (v1)

Currently, `smart_search` is an alias for `hybrid_search` with `rerank=False`.

Future versions (v2) will classify queries:
- Short, exact terms → keyword
- Natural language questions → semantic
- Mixed → hybrid
- Code snippets → keyword with syntax parsing

### Example Usage

```python
# Simple auto-detection
smart_search("FastAPI websocket patterns")

# With stack filter
smart_search(
    "authentication best practices",
    stack="nextjs,typescript"
)

# Specify top_k
smart_search("React hooks", top_k=10)
```

## Performance Comparison

| Mode | Avg Latency | Token Cost | Best Precision Scenario |
|------|-------------|------------|------------------------|
| `search_knowledge` | 50ms | 0 | Exact term lookup |
| `semantic_search` | 450ms | ~100 tokens | Conceptual queries |
| `hybrid_search` | 600ms | ~100 tokens | Mixed queries |
| `hybrid_search` (rerank) | 950ms | ~100 tokens | Critical queries |
| `smart_search` | 600ms | ~100 tokens | General use |

<Note>
Latencies measured on MidOS cloud (670K vectors, 46K chunks). Self-hosted performance depends on hardware and LanceDB configuration.
</Note>

## Choosing the Right Mode

<Steps>
  <Step title="Start with hybrid">
    For most queries, `hybrid_search` gives the best quality/speed tradeoff.

    ```python
    hybrid_search("your query here")
    ```
  </Step>

  <Step title="Use keyword for exact lookups">
    If you know the exact term or file name:

    ```python
    search_knowledge("specific_function_name")
    ```
  </Step>

  <Step title="Use semantic for concepts">
    If the query is a natural language question:

    ```python
    semantic_search("how to prevent SQL injection")
    ```
  </Step>

  <Step title="Add stack filtering">
    For all modes, filter by technology stack:

    ```python
    semantic_search("caching patterns", stack="redis,python")
    ```
  </Step>

  <Step title="Enable reranking for critical queries">
    For high-stakes decisions, enable cross-encoder reranking:

    ```python
    hybrid_search("security best practices", rerank=True)
    ```
  </Step>
</Steps>

## Troubleshooting

<AccordionGroup>
  <Accordion title="No results with semantic_search">
    **Cause**: Query too short or too generic.

    **Fix**: Add more context.

    ```python
    # Too vague
    semantic_search("testing")

    # Better
    semantic_search("React component testing with Vitest and Testing Library")
    ```
  </Accordion>

  <Accordion title="Keyword search misses obvious results">
    **Cause**: Content uses synonyms or different phrasing.

    **Fix**: Switch to semantic or hybrid.

    ```python
    # Keyword misses synonyms
    search_knowledge("speed up queries")  # Won't find "optimize performance"

    # Semantic finds conceptual matches
    semantic_search("speed up queries")
    ```
  </Accordion>

  <Accordion title="Hybrid search is slow">
    **Cause**: Reranking adds 300-500ms overhead.

    **Fix**: Disable reranking for exploratory search.

    ```python
    # Fast hybrid (no rerank)
    hybrid_search("query", rerank=False)
    ```
  </Accordion>

  <Accordion title="Stack filtering not working">
    **Cause**: Stack tokens not matching content.

    **Fix**: Use canonical names.

    ```python
    # Wrong: abbreviations
    semantic_search("auth", stack="js,pg")

    # Correct: full names
    semantic_search("authentication", stack="javascript,postgresql")
    ```
  </Accordion>
</AccordionGroup>
