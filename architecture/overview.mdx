MidOS is a specialized MCP (Model Context Protocol) server that provides curated, validated developer knowledge to AI coding agents. This page covers the high-level system architecture.

## FastMCP Server

The core server is implemented in `modules/mcp_server/midos_mcp.py` using [FastMCP](https://github.com/jlowin/fastmcp), a modern framework for building MCP servers.

### Transport Modes

MidOS supports **dual transport**:

- **stdio** (default): Standard input/output for local CLI tools like Claude Code
- **streamable-http**: HTTP-based transport for remote connections and web clients

**Running the server:**

```bash
# stdio mode (default, backward compatible)
python -m modules.mcp_server.midos_mcp

# HTTP mode (remote, streamable)
python -m modules.mcp_server.midos_mcp --http --port 8419
```

### MCP Tools (18 total)

The server exposes 18 tools organized by tier:

**Community Tier (free, no API key required):**
- `search_knowledge` - Keyword search across all knowledge layers
- `list_skills` - Browse 200+ skill packs with optional stack filtering
- `get_skill` - Get complete skill document by name
- `get_protocol` - Get protocol/pattern documentation
- `hive_status` - System health and statistics
- `project_status` - Live dashboard + quick-start guide
- `agent_handshake` - Personalized onboarding
- `agent_bootstrap` - Generic onboarding (deprecated)

**Dev Tier (requires API key):**
- `semantic_search` - Vector search using LanceDB + Gemini embeddings
- `get_eureka` - Access EUREKA breakthrough documents
- `get_truth` - Get truth patch documents
- `research_youtube` - Queue videos for research
- `chunk_code` - AST-based code chunking
- `memory_stats` - Vector store statistics
- `pool_status` - Multi-instance coordination status
- `episodic_search` - Search past agent experiences

**Admin Tier:**
- `episodic_store` - Store new episodic memories
- `pool_signal` - Signal to multi-instance pool

## Knowledge Storage

The knowledge base is organized in the `knowledge/` directory with a **5-layer quality pipeline**:

```
staging/ → chunks/ → skills/ → truth/ → EUREKA/ → SOTA/
 (entry)    (L1)      (L2)      (L3)     (L4)      (L5)
```

See [Knowledge Layers](/architecture/knowledge-layers) for detailed explanation.

### Directory Structure

```
knowledge/
├── chunks/     # L1: 88 curated knowledge files
├── skills/     # L2: 78 organized skill directories
├── truth/      # L3: 2 empirical patches (50 items total in production)
├── EUREKA/     # L4: 2 catalog files (383 items in production)
└── SOTA/       # L5: Best-in-class patterns (140 items in production)
```

Reference: `modules/mcp_server/midos_mcp.py:79-86`

## Vector Store

MidOS uses **LanceDB** with **Gemini embeddings** for semantic search:

- **Embedding model**: `gemini-embedding-001` (3072 dimensions)
- **Search modes**: Vector-only, keyword-only (BM25), or hybrid (RRF fusion)
- **Indexing**: Full-text search (FTS) index on text column
- **Caching**: Query embedding cache (5-minute TTL, LRU eviction)
- **Memory decay**: Importance-weighted exponential decay for knowledge lifecycle

See [Vector Store](/architecture/vector-store) for implementation details.

## Performance Characteristics

### Search Latency

- **Keyword search**: ~50ms (direct file scanning)
- **Vector search**: ~200ms (LanceDB query + embedding)
- **Hybrid search**: ~250ms (RRF fusion of both)
- **Cached query**: ~1ms (query embedding cache hit)

### Embedding Performance

- **Original**: ~1200ms/text (sequential API calls)
- **Batch only**: ~35ms/text (50 texts per call)
- **Batch + concurrent**: ~19ms/text (4 workers)
- **Cache hit**: ~0ms (no API call)
