# EUREKA System

The EUREKA system is **Layer 4** in MidOS's [Knowledge Pipeline](/concepts/knowledge-pipeline). EUREKA documents represent validated breakthrough insights with **measurable ROI** or **Kaizen improvements**.

## What is a EUREKA?

A EUREKA is:
- **Validated**: Tested in production with metrics
- **Measurable**: Quantified improvements (%, ms, cost, etc.)
- **Breakthrough**: Not incremental—represents a discovery or optimization
- **Reproducible**: Others can achieve similar results

> "EUREKA" = "We found the optimal way to do X"

A EUREKA is **NOT**:
- "Something interesting I read" → That's a **chunk**
- "A pattern that works" → That's a **skill**
- "A verified fact" → That's a **truth**

## Promotion Criteria

A knowledge item becomes a EUREKA when it meets **≥1** of these criteria:

### 1. Measurable ROI Improvement

Quantifiable performance, cost, or quality gains:

```
Before: 10,000 tokens per request
After:  3,000 tokens per request
ROI:    70% reduction

→ EUREKA: SKILLS_CLI_TOKEN_OPTIMIZATION
```

### 2. Kaizen Optimization

Comprovable workflow improvement with metrics:

```
Before: Sequential embedding (1200ms per text)
After:  Batch + concurrent (19ms per text)
Kaizen: 63x speedup

→ EUREKA: BATCH_EMBEDDING_OPTIMIZATION
```

### 3. Discovered Optimal Approach

Found the best way to work with a technology:

```
Tested: Vector-only, Keyword-only, Hybrid (RRF)
Winner: Hybrid search
Metric: +9.3% relevance on 9,753 vectors

→ EUREKA: EUREKA_HYBRID_SEARCH_RRF_2026
```

## Validation Process

### Phase 1: ROI Measurement

1. **Establish baseline**: Measure current performance
2. **Implement change**: Deploy optimization
3. **Measure improvement**: Collect post-change metrics
4. **Calculate ROI**: Quantify improvement

```python
# Example: Token optimization validation
baseline_tokens = measure_tokens(use_mcp_direct=True, n=100)
optimized_tokens = measure_tokens(use_skills_cli=True, n=100)
roi_percent = (baseline - optimized) / baseline * 100

if roi_percent >= 10:  # 10% minimum improvement
    status = "VALIDATED"
```

### Phase 2: Test Suite

Every EUREKA requires passing tests:

```python
class TestEureka:
    def test_baseline_scenario(self):
        """Test original problem exists."""
    
    def test_solution_works(self):
        """Test solution solves problem."""
    
    def test_edge_cases(self):
        """Test boundary conditions."""
    
    def test_production_load(self):
        """Test under production-like load."""
    
    def test_rollback(self):
        """Test graceful degradation."""
```

### Phase 3: Production Validation

- **Minimum runtime**: 7 days in production
- **Monitoring**: Metrics tracked continuously
- **Incident count**: Must not introduce regressions
- **Rollback plan**: Documented and tested

### Phase 4: Peer Review

- Technical review by 2+ engineers
- Metrics validation by independent reviewer
- Documentation completeness check
- Production readiness checklist

## Degradation Triggers

A EUREKA is degraded back to Truth/Skill when:

- **New approach discovered**: Better solution with superior ROI
- **Metrics disproven**: Independent validation fails to reproduce
- **Context changed**: Stack version makes pattern obsolete
- **Production failure**: Causes incidents in production

## ATOM Pattern

ATOMs are micro-EUREKAs—small, atomic optimizations:

```markdown
# ATOM-001: Cache Key Normalization

**ROI**: 12% cache hit rate improvement

**Pattern**:
\`\`\`python
def normalize_cache_key(query: str) -> str:
    return query.lower().strip()
\`\`\`

**Metric**: Cache hit rate 0.65 → 0.73 (+12%)
```

## Best Practices

### For EUREKA Authors

1. **Measure everything**
   - Baseline before optimization
   - Multiple data points (not single test)
   - Statistical significance (n ≥ 30)

2. **Document the journey**
   - What didn't work
   - Why you tried this approach
   - Lessons learned

3. **Include reproduction steps**
   - Exact versions tested
   - Test data/fixtures
   - Expected results

4. **Be honest about limitations**
   - What scenarios it doesn't cover
   - Trade-offs made
   - When NOT to use this pattern

5. **Provide rollback plan**
   - How to revert if it fails
   - Graceful degradation
   - Feature flags

### For EUREKA Consumers

1. **Verify compatibility**
   - Check versions match your stack
   - Review production considerations
   - Test in staging first

2. **Measure independently**
   - Don't trust claimed ROI blindly
   - Measure in your own context
   - Compare apples-to-apples

3. **Adapt to your context**
   - EUREKAs are proven patterns, not copy-paste
   - Consider your scale, constraints
   - May need adjustments

4. **Report results**
   - Share your metrics (confirm or refute)
   - File issues if patterns don't work
   - Contribute improvements
